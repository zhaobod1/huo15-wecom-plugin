# v2.8.1 (2026-04-22)

## 🛡️ 修复：拒绝 `@all` 等广播字面量目标

### 背景

历史事故：主 Agent 在群聊场景里把大文件（图片/视频/Word）误用 `touser: "@all"` 调用企业微信自建应用 API，导致私聊文件被广播到全公司可见范围。

事故路径并不在插件内部（outbound 短路链和 `delivery-service` 已经拦住了 chatId 走 Agent API 的错误路径），而是 Agent LLM 在工作区里自己写脚本 (`send_wecom_file.py`) 时把 `@all` 当成"默认收件人"传了进来。插件层之前对裸字符串 `@all` 不做检查，直接当 `touser` 放行。

### 改动

在 `resolveWecomTarget` 入口加**广播字面量闸门**（`src/target.ts`）：

- 新增 `BROADCAST_LITERALS = {"@all", "@everyone", "*", "all"}`
- 新增导出 `isBroadcastLiteral(raw)` —— 供上层需要提前判断时使用
- `resolveWecomTarget("@all")` 现在返回 `undefined`，让 `WecomAgentDeliveryService.resolveTargetOrThrow` 抛出清晰错误
- 同时拦截脱壳形式：`wecom:user:@all` / `user:@all` / `party:@all` / `wecom-agent:@all` 都会被 `strippedForCheck` 二次判断拦下
- 命中时打印 `[wecom-target] refusing broadcast literal target=…` 警告日志，便于追溯

如果确实需要对可见范围内全员推送（极少见），调用方必须使用显式前缀 `party:<id>` / `tag:<id>` 并自己承担范围确认责任。

### 新增测试

`src/target.test.ts`：
- 广播字面量识别（含大小写 & 脱壳形式）
- `resolveWecomTarget` / `resolveScopedWecomTarget` 双路径拒绝
- 正常 userid / party / chatid 目标不受影响

### 兼容性

- 仅拒绝字面量 `@all`/`@everyone`/`*`/`all`（大小写不敏感），不影响任何真实 userid
- 上层 `delivery-service.ts` 依赖 `resolveTargetOrThrow` 抛错，原有行为保持一致
- 无配置变更，升级即生效

# v2.8.21 (2026-05-04)

**诊断与韧性增强：reply.ts MEDIA: parser 加 detected/warn log + 86008 群权限识别 + GUIDANCE 加 ✅/❌ 示例对比**

## 背景

v2.8.20 让 reply.ts 接管 MEDIA: 字面量后，赵博又试发 zip 仍然收不到附件。另一会话 Claude 诊断时找到关键证据：

```
[wecom-agent-api] sendText response ... errcode=86008
errmsg=you has no privilege to access this chat, which is created by other agent.
```

但同时它又说 "extractMediaDirectives() 从未被调用，MEDIA: 行从来没被解析到" —— 是因为 v2.8.20 我**只在 outbound.ts 加了 console.log**（`[wecom-outbound] MEDIA directive(s) detected`），reply.ts 那边没加 log 标志。下游 Claude grep 不到 outbound 的 log 就误以为 parser 完全没工作。

实际两个独立问题混在一起：

1. **没法验证 reply.ts parser 触发**：v2.8.20 只在 outbound.ts 加了 detected log，reply 路径无 log
2. **86008 群权限错误**：机器人在某个群里没有"发媒体"权限（群是其他企微应用创建的、本机器人不是 owner）
3. **MEDIA: 必须独立成行**的边界：LLM 偶尔会把 `MEDIA:` 嵌入正文中部（如 `📎 MEDIA: ~/foo.zip`），parser 不匹配但用户/我们看不到

v2.8.21 一次性修这三件事的诊断 + 韧性。

## 改动

### 1. `src/transport/bot-ws/reply.ts` — 加诊断 log（核心）

reply.ts deliver 入口的 `extractMediaDirectives` 调用三态：

| 状态 | 触发 | 行为 |
|---|---|---|
| ① 没 emit MEDIA: | text 里压根没 "MEDIA:" 子串 | 沉默（正常） |
| ② emit 了 + 抽出 | `MEDIA:` 单独成行匹配 | `console.log` `[wecom-ws] MEDIA directive(s) detected via reply.deliver (count=N, kind=block/final, peer=group:xxx/direct:xxx, paths=[...])` |
| ③ emit 了但格式错 | text 含 "MEDIA:" 子串但行级匹配未命中 | `console.warn` `[wecom-ws] MEDIA: substring present but no directive line matched ... LLM 必须把 'MEDIA: <path>' 单独成行` + 抽样问题行展示 |

之前 v2.8.20 没区分这三态，gateway.log 看不到任何 parser 信号。本版让运维/未来诊断会话能立刻看出"MEDIA: 是没 emit / 抽出来了 / 写错格式"。

### 2. `src/transport/bot-ws/media.ts` — errcode 详细 log + 86008 专项识别

`uploadAndReplyBotWsMedia` 失败的 catch 块：抽出 errcode/errmsg/raw 打 log。86008 走专门的 warn 文案（"机器人不是该群的'创建/可发媒体'应用"）+ 提示"自动走 share-fallback 链接发送"——既不让用户/运维看到神秘的"上传失败"，又主动告诉他们后续行为。

`trySendShareFallbackOnReject` 内部 `sendMessage` 失败的 catch 块：同样的 errcode 抽取。如果 sendMessage 也撞 86008（说明 replyMedia 和 sendMessage 在该群都被拒），警示降级到 "把链接塞进 finalText 走 replyStream 文本回复"路径——文本被动回复权限模型不同，能 work（用户实测群里 @机器人发文本一直能收到回复）。

成功路径也加了一条 log：`[wecom-ws] share-fallback 链接已发送到 <chatId>（X.XMB → <url>）`，便于判断"链接发出去了 vs 用户没看到"。

新增两个内部 helper：

- `describeWecomError(error)` — 拍平 SDK reject error 的 errcode/errmsg/raw，便于复用
- `isAgentChatPrivilegeError(error)` — 检测 errcode 86008（不可重试的群权限错误）

注释里枚举关键 errcode：86008 / 84080 / 60011 / 845xx，便于未来诊断别的错误时复用。

### 3. `index.ts` — `WECOM_BOT_WS_MEDIA_GUIDANCE` 加 ✅/❌ 对比示例

之前文案只列要求，不给反例。本版加：

**✅ 正确示例**：
```
你要的文件已经准备好。

MEDIA: ~/.openclaw/media/outbound/report.pdf
```

**❌ 错误示例**（parser 抽不出来，warn log 会提示）：
- `📎 MEDIA: ~/foo.zip` — 行首有 emoji
- `请查收 MEDIA: ~/foo.zip` — 行首有正文
- `MEDIA: ~/foo.zip 已发送` — 路径后有正文
- `> MEDIA: ~/foo.zip` — markdown 引用 / 列表前缀
- `"MEDIA: ~/foo.zip"` — 整体被引号包住

也加了"客观限制"段落明确企微规则：

- 文件 > 上限 → share-fallback（落盘 + 公网下载链接）
- 群是其他企微应用创建的 → 86008，自动降级到 share-fallback 链接（仍能在群里看到下载地址）
- 决策提示：群里 ≤ 20MB 优先 MEDIA:（直接附件最佳 UX），> 20MB 或不确定 → `enhance_share_file`

### 4. `src/transport/bot-ws/reply.test.ts` — 新增 2 个 v2.8.21 诊断 log 测试

- `v2.8.21: logs '[wecom-ws] MEDIA directive(s) detected' when parser successfully extracts` — 用 `vi.spyOn(console, 'log')` 验证 detected log 被打 + 含 count/kind/peer 字段
- `v2.8.21: warns when 'MEDIA:' substring present but no directive line matched (LLM 写错格式)` — 用 LLM 错误示例 `📎 MEDIA: /tmp/foo.zip 已经准备好啦` 触发 warn log + 验证 upload **不**被调用（parser 没匹配 → 不发 media）

reply.test.ts 现 24/24 通过（v2.8.17 progressMode 6 + v2.8.20 MEDIA 3 + v2.8.21 log 2 + 老的 13）。

## 现在 gateway.log 长什么样

正常 zip 发送：

```
[wecom-ws] MEDIA directive(s) detected via reply.deliver (count=1, kind=final, peer=group:wrXXX, paths=["~/foo.zip"])
[wecom-ws] share-fallback 链接已发送到 wrXXX（25.3MB → https://keepermac.huo15.com/plugins/enhance-share/abc-foo.zip）
```

或文件 ≤ 20MB 直接附件成功（无 share-fallback 那一行）。

LLM 写错格式（嵌入正文）：

```
[wecom-ws] MEDIA: substring present but no directive line matched (kind=final, peer=group:wrXXX, sample="📎 MEDIA: /tmp/foo.zip 已经准备好啦"). LLM 必须把 'MEDIA: <path>' 单独成行...
```

群权限拒绝：

```
[wecom-ws] uploadAndReplyBotWsMedia 86008 群权限错误：当前机器人不是该群的"创建/可发媒体"应用 (file=foo.zip type=file errmsg=you has no privilege to access this chat ...)。自动走 share-fallback 链接发送。
[wecom-ws] share-fallback sendMessage 也撞 86008 群权限 (chatId=wrXXX, errmsg=...)。降级把链接塞进 finalText，由 reply.ts replyStream 文本回复发出。
```

未来运维 grep `MEDIA directive\|share-fallback\|86008` 就能完整看到媒体发送的状态机。

## 经验沉淀（写进 changelog）

**任何"我加了 hook/parser 但效果不明"的诊断盲点**：必须在 hook 入口加 `console.log` 标志（含触发计数 + 关键参数）。光在文档 / changelog 写"v2.8.X 加了 parser"是不够的——下次故障下游会话 grep gateway.log 看不到信号就会误判"parser 没工作"。

v2.8.20 → v2.8.21 这一跳就是这个教训：v2.8.20 reply.ts parser 实际可能在工作（mock 单测过），但 gateway.log 沉默 → 下游 Claude 诊断错方向。本版补上 detected/warn 两态 log，未来类似事故能立刻定性。

参考 memory `feedback_tool_description_vs_prompt_level_constraint.md` —— GUIDANCE 加 ❌ 反例比泛泛的"必须独立成行"有效得多。

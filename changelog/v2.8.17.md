# v2.8.17 (2026-05-02)

**长任务结果回流修复 + 进度反馈模式可配置（progressMode）**

## 背景

用户反馈：在 `agent:main:wecom:direct:zhaobo` 这类 direct 会话里，发起耗时任务（如 LLM 长 reasoning + 多轮 tool call）后，企微一直停在"⏳ 正在思考中..."占位，**最终结果永远不显示**——但 OpenClaw UI 同一会话里能看到完整结果。

排查到根因在 [`src/transport/bot-ws/reply.ts`](src/transport/bot-ws/reply.ts) 的 final delivery catch 路径：当任务时长超过约 2 分钟、企微服务端把 reqId 释放（errcode 846605 invalid req_id）或 stream message update window 关闭（errcode 846608）后，`replyStream(...,true)` 抛 `isInvalidReqIdError` / `isExpiredStreamUpdateError` —— 而旧代码遇到 `isTerminalReplyError` 直接 `params.onFail?.(error); return;`，**完全跳过了 sendMessage 主动推送和 Agent API 两层 fallback**，结果就丢在了插件层。

## 修复 1：reqId 失效 / 流过期降级到主动推送（核心 P0）

[`src/transport/bot-ws/reply.ts`](src/transport/bot-ws/reply.ts) deliver `kind:'final'` 路径：

```diff
- } else if (isTerminalReplyError(error)) {
-   params.onFail?.(error);
-   return;
+ const isReplyChannelDead =
+   isInvalidReqIdError(error) || isExpiredStreamUpdateError(error);
+ if (isWsTimeout) { ... }
+ else if (isReplyChannelDead) {
+   console.warn(`[wecom-ws] reply channel closed (${msg}) for ${peerId}, trying active push fallback`);
+   // 不再 onFail 短路；fall through 到 sendMessage / Agent API 两层 fallback
  }
```

逻辑：SDK 的 `replyStream` 通道（绑定 inbound req_id）虽然死了，但 `sendMessage` 主动推送通道是**独立**的，仍能把已生成内容送出去。降级链路不变：

1. **Tier 1**: Bot WS active push（`client.sendMessage`）— 直接给当前 chat 发主动消息
2. **Tier 2**: Agent API（`fallbackAgentApiDelivery`）— 最后兜底（要求该账号配了 `agent.apiConfigured`）

Test 替换：旧的 `swallows expired stream update errors during delivery`（断言 `onFail` 被调）改成新的 `falls back to active push when reply channel is closed by ... during final delivery` 用 `it.each` 同时覆盖 invalid req_id 和 expired stream，断言：
- `replyStream` 被调用 1 次（先尝试，失败）
- `sendMessage` 被调用（fallback 主动推送）
- `onDeliver` 被调用、`onFail` **不**被调用（这就是修复要解决的问题）

## 修复 2：新增 `progressMode` 配置项（4 选项，默认升级）

旧实现下，长任务期间用户只看到固定的"⏳ 正在思考中..."循环（每 3s 重发，120s 后停），没有任何"任务还活着"的进度感。本版新增配置：

```yaml
# 配置位置：channels.wecom.[bot|accounts.<id>.bot].progressMode
channels:
  wecom:
    accounts:
      default:
        bot:
          progressMode: progress     # 默认（v2.8.17 升级）
          progressDelayedMs: 30000   # 仅 delayed 模式生效
```

| 模式 | 行为 |
|---|---|
| **`progress`**（**默认**） | 阶段化心跳文案，按已等待时长升档：0-30s "⏳ 正在思考中..." → 30-60s "⏳ 仍在处理中（已 Xs）..." → 60-120s "⏳ 任务较复杂（已 Xm），请稍候..." → 120s 临界停 keepalive 前发一条 "⏳ 任务仍在执行（已 Xm+），完成后会主动推送结果" 让用户安心 |
| **`heartbeat`** | v2.8.16 及之前的固定 "⏳ 正在思考中..."，每 3s 重发，120s 后停（向后兼容用） |
| **`delayed`** | 默认沉默；`progressDelayedMs`（默认 30000ms）后单次发提示，**不循环**（适合"宁愿沉默不要打扰"的用户） |
| **`off`** | 完全不发占位（高级用户用，依赖 LLM partial chunk 自身可见性） |

显式传 `placeholderContent` 优先级最高（向后兼容老 config），覆盖所有 progressMode 升档逻辑。

实现要点：
- `buildProgressPlaceholder(elapsedMs)` module-level 纯函数，便于单测
- `delayed` 模式跳过初始 `sendPlaceholder()`，单 `setTimeout(progressDelayedMs)` 触发后不再循环
- `off` 模式跳过整个 keepalive 注册分支
- 透传链路：openclaw config (`channels.wecom.bot.progressMode`) → `WecomBotConfig` → `resolveBotAccount` → `botAccount.config.progressMode` → `BotWsSdkAdapter.start.handleFrame` → `createBotWsReplyHandle` params

## 兼容性

完全向下兼容。

- 老 config（无 `progressMode`）走默认 `progress` 模式 → 阶段化文案，无 breaking change（行为是"显示更多信息"，没有少东西）
- 显式 `placeholderContent` 仍然 override 一切 progressMode 升档（老用户的自定义文案保留）
- `MAX_PARTIAL_REPLIES = 8` / `MAX_KEEPALIVE_MS = 120s` / ack timeout watchdog 阈值不变
- bot-webhook 通道未改动（v2.8.14 已有独立的 long-task rescue 机制）

## 不变 / 不在本次范围

- **A "phases" 模式（"🔧 正在调用 file_read..."）未实现**：需要 OpenClaw runtime 暴露 `tool:start/tool:end` 公共 hook 才能拿到工具名；纯 wecom 拿不到，强行做就是猜。等 runtime 加 hook 后单独发版。
- **C "stream" 模式（提高 partial 上限）未实现**：当前 `MAX_PARTIAL_REPLIES = 8` 是为防 SDK ack 队列饱和的熟知折中；动这个有风险，且不解决用户报的"final 不来"根因（核心 bug fix 已彻底解决）。
- **零跨插件耦合**：本次实现完全在 `@huo15/wecom` 内；不引入 enhance import；不依赖 enhance 文件系统契约（share-fallback 是已有的、enhance 可装可不装）；wecom 单装即可用 v2.8.17 全部新功能。

# v2.8.19 (2026-05-04)

**outbound.sendText 实装 "MEDIA: <path>" 指令解析 — 修群聊里发文件失败的根因事故**

## 背景

群会话 `agent:main:wecom:direct:zhaobo` 之外的另一个群 `agent:main:wecom:group:wrgzumeqaadcsaffbvgfobppv-_6ccwg` 里，赵博让 `@贾维斯 打包zip发给我`，trajectory 复盘：

1. LLM 按 `index.ts` 的 `WECOM_BOT_WS_MEDIA_GUIDANCE` 引导（注入到 system context）正确生成了 `MEDIA: ~/.openclaw/media/outbound/huo15-rustdesk-deploy.zip` 单行指令
2. zip 文件确实在 outbound 目录（7754 字节，文件正常）
3. **但 outbound 完全没有解析这个字面量** — `grep -nE "MEDIA:" --include="*.ts" src/` 命中的全是 GUIDANCE 文本本身，没有任何 parser
4. → "MEDIA: ..." 行当成普通文本发了出去，群里收到一行字 "MEDIA: ~/..."（用户根本看不见，企微对长得像路径的文本会折叠）
5. 用户问"怎么没发"，LLM 又重复 emit 一次 `MEDIA:` 行 — 还是没下文

GUIDANCE 是从 v2.8.x 早期就有的 dead prompt — 文档说"必须用 MEDIA: 指令"，但实现一直没补。**v2.8.19 把这条文档兑现**。

## 改动

### 1. `src/outbound.ts` — 新增 `extractMediaDirectives(text)` 解析器

按行扫描，命中 `^\s*MEDIA:\s*(.+?)\s*$` 的整行抽出来作为媒体路径返回；剩余文字组装回 `residualText`。自动处理：

- trim 前后空白
- 去掉成对引号（`"..."` / `'...'`）
- 展开 `~/` 到当前用户 home（与 sendMedia 内部 `mediaUrl.startsWith("~")` 行为一致）
- 大小写不敏感（`media:` / `Media:` / `MEDIA:` 都接受）
- 多行 `MEDIA:` 共存按出现顺序全部抽出

**关键设计**：必须**整行单独**才匹配，避免误伤普通文本里出现的 "MEDIA:" 子串（比如 "参数 MEDIA: foo 是干啥的？"）。

### 2. `src/outbound.ts` — `sendMedia` body 抽出 `sendMediaInternal()` internal function

为了 `sendText` 的 MEDIA: 解析路径能复用三层（kefu/botWs/agent）fallback 链，不重复实现。

`wecomOutbound.sendMedia` 现在直接 `= sendMediaInternal`，外部 API 不变。

### 3. `src/outbound.ts` — `sendText` 入口集成 MEDIA: 解析

`sendText` 入口先 `extractMediaDirectives(ctx.text)`：

- 命中 mediaPaths → 循环逐个调 `sendMediaInternal({...ctx, mediaUrl, text: ""})`
- 单个 media 失败**不吞整批**：记 `console.warn`，继续发剩下的 + residualText
- residualText 为空且发了 media → 直接返回最后一次 media 的 messageId
- 否则 residualText 走原 sendText 文本路径（kefu / botWs / agent，含 markdown image 抽取）

### 4. `index.ts` — `WECOM_BOT_WS_MEDIA_GUIDANCE` 文案强化

之前文案只说"必须用 MEDIA: 指令"。新版加：

- 明确 v2.8.19+ 真正生效
- 列出有效路径形式（绝对路径 / `~/` 自动展开）+ 示例 zip
- 强调"任意文件大小都用本指令发"——杜绝 LLM 自创 `📎` / `FILE:` 等字面量
- 多行 MEDIA: 行为说明（按序、单个失败不影响后续）
- 大文件（>20MB）建议改用 `enhance_share_file` 出公网下载链接

### 5. `src/outbound.test.ts` — 新增 9 个单测覆盖 `extractMediaDirectives`

覆盖：empty/null、no directive、单行抽取、`~/` 展开、引号去除、大小写不敏感、多行顺序、in-line 文本不误伤、空白容忍、空 path 跳过。

## 配套 enhance v6.1.3（同日发布）

同日发布的 `@huo15/huo15-openclaw-enhance@6.1.3` 把 `enhance_share_file` 工具的 prompt supplement 也修了：去掉「≥10MB」阈值，强制小文件也走 share file（链接路径）。两边修复路径互补：

- **wecom 渠道直发文件 (附件进群)** → MEDIA: 指令 / 本插件
- **大文件或跨渠道分享 (公网链接)** → enhance_share_file / enhance v6.1.3+

LLM 拿到 GUIDANCE + enhance prompt supplement 双引导后能正确选路径。

## 兼容性

完全向下兼容。

- 老 prompt / 老 LLM 输出**不含** `MEDIA:` 行 → 行为完全不变（`extractMediaDirectives` 返回 `mediaPaths: []`，原 sendText 路径走完）
- `wecomOutbound.sendMedia` 公开 API 签名 + 行为不变（直接 = sendMediaInternal）
- markdown image (`![alt](url)`) 抽取路径**不动** — 跟 MEDIA: 指令并存
- `MEDIA:` 出现在普通文本中部（不在行首）不抽取 — 避免误伤
- `extractMediaDirectives` 是 `export function`，便于上层调用 / 单测复用，但不是 channel adapter API 的一部分（不破坏 OpenClaw plugin SDK contract）

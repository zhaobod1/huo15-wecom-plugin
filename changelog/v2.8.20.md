# v2.8.20 (2026-05-04)

**v2.8.19 hotfix：bot-ws/reply.ts deliver 入口也接管 MEDIA: —— 群聊 @机器人触发的 in-context reply 走 reply 路径不走 outbound.sendText**

## 背景

v2.8.19 在 `outbound.sendText` 加了 MEDIA: parser，自信地宣布"修了群里发 zip 失败事故"。装上、重启 gateway 后，赵博立刻在群里再试 `@贾维斯 huo15-rustdesk-deploy打包发给我`：

- LLM 按新 GUIDANCE 完美 emit 了 `"📎\n\nMEDIA: ~/.openclaw/media/outbound/huo15-rustdesk-deploy.zip"`
- 但群里**仍然没收到 zip 附件**
- gateway log 里**没有任何** `[wecom-outbound] MEDIA directive(s) detected` 日志（我加的 console.log 完全没触发）

### 排查到的真根因

```
2026-05-04T18:41:10.003+08:00 [wecom-runtime] deliver-start ... transport=bot-ws kind=block
2026-05-04T18:41:10.004+08:00 [wecom-ws] Reply message sent via WebSocket, reqId: 3sIa5YFJQ...
2026-05-04T18:41:10.005+08:00 [wecom-runtime] outbound ... summary="bot-ws 📎"
```

**关键**：log 行 `[wecom-ws] Reply message sent via WebSocket` 来自 SDK 的 `replyStream`（in-context reply），是 **bot-ws/reply.ts 的 deliver()** 入口。而 `outbound.sendText` 是给 **Agent 主动推送**用的（比如 cron 任务推消息给用户），完全不是同一条链路。

群聊 @机器人 / direct DM 之类的 inbound message 触发的 LLM 回复**永远走 reply 路径**，不会经过 outbound.sendText —— 我 v2.8.19 修的 outbound parser 完全不会被调用。

```
inbound user message
  → BotWsSdkAdapter.handleFrame
    → createBotWsReplyHandle.deliver(payload, kind)   ← reply.ts，群消息回复走这里
      → replyStream / uploadAndReplyBotWsMedia
                                                       ← outbound.sendText 完全不在调用链上
```

## 改动

### 1. `src/transport/bot-ws/reply.ts` — deliver 入口加 MEDIA: parser

```diff
+ import { extractMediaDirectives } from "../../outbound.js";

  deliver: async (payload: ReplyPayload, info) => {
    notifyPeerActive();
    if (payload.isReasoning) { ... return; }

+   // v2.8.20 ⭐ 解析 LLM emit 的 "MEDIA: <path>" 字面量（与 outbound.sendText 同源）
+   let normalizedText = payload.text ?? "";
+   const directiveMediaPaths: string[] = [];
+   if (normalizedText) {
+     const ex = extractMediaDirectives(normalizedText);
+     if (ex.mediaPaths.length > 0) {
+       normalizedText = ex.residualText;
+       directiveMediaPaths.push(...ex.mediaPaths);
+     }
+   }
+
-   const text = payload.text?.trim() || "";
+   const text = normalizedText.trim() || "";
-   const incomingMediaUrls = payload.mediaUrls || (payload.mediaUrl ? [payload.mediaUrl] : []);
+   const incomingMediaUrls = [
+     ...(payload.mediaUrls || (payload.mediaUrl ? [payload.mediaUrl] : [])),
+     ...directiveMediaPaths,
+   ];
    // ... 原 mediaUrls / final 处理逻辑接管
  }
```

抽出的 mediaPaths 合并到 `incomingMediaUrls` —— reply.ts 现有的 `final` 路径会调 `uploadAndReplyBotWsMedia({wsClient, frame, mediaUrl, ...})` 走 SDK 的 `aibot_respond_msg` **被动回复通道**（绑定 inbound reqId）。这正是 zhaobo 群聊场景所需的"群里直接收到附件"路径。

block 路径里 mediaPaths 也会被 `mergeDeferredMediaUrls` 累积，final 时统一发出，符合 reply.ts 的 streaming 语义。

### 2. `src/outbound.ts` extractMediaDirectives 已 export — 直接复用

v2.8.19 时已经 `export function extractMediaDirectives(...)`，本版 reply.ts 直接 import，零代码重复。

### 3. `src/transport/bot-ws/reply.test.ts` — 新增 3 个 v2.8.20 MEDIA: 接管测试

- `extracts MEDIA: directive from final text and triggers media upload via reply channel` — 单 MEDIA: 行触发 `uploadAndReplyBotWsMedia` + 残余文本 "📎" 走 replyStream
- `MEDIA: directive in block payload defers media to final, accumulates correctly` — block 阶段 MEDIA: 进 deferredMediaUrls 累积，final 统一发
- `multiple MEDIA: directives in one final payload all get uploaded` — 一次 emit 三个 MEDIA: 行，三个 `uploadAndReplyBotWsMedia` 全部按序触发

reply.test.ts 现 22/22 通过（含 v2.8.17 的 6 个 progressMode + v2.8.20 的 3 个 MEDIA: 接管）。

## 配套 v2.8.19 的 outbound.sendText 修复仍然有用

v2.8.19 在 outbound 加的 MEDIA: parser **不是白做** —— 用于：
- Agent 主动推送场景（cron 任务、定时通知、跨会话主动 send）
- 跨渠道 routing 走 wecomOutbound.sendText
- 文档化 outbound channel adapter 的能力

v2.8.20 reply 路径修复了**用户实际触发的入站 → LLM 回复**这条主线。两条路径的 parser 同源（都走 `extractMediaDirectives`），行为一致。

## 经验沉淀（写进 memory）

**bug 报告流程的盲点**：v2.8.19 时只看 `extractMediaDirectives` 是否被定义和单测通过就发版了，**没在集成层验证哪条路径真实触发**。`bot-ws/reply.ts` 是 in-context reply（绑 reqId，群聊 / DM 入站消息回复），`outbound.sendText` 是 Agent 主动推送（cron、跨会话 send_message 工具调用）—— 两条独立链路，名字相似但调用上下文截然不同。

未来类似"hook/parser 加了但功能不生效"调试 SOP：

1. 在 hook/parser 入口处加显式 `console.log` 标记（如 `MEDIA directive(s) detected`）
2. 重启 gateway 后让用户立刻复现
3. **grep gateway log 看标记是否出现** —— 不出现 = 你修的不是真路径
4. 如果不出现，从 trajectory + log 反推真实调用链（看 `[wecom-runtime] deliver-start` / `Reply message sent via WebSocket` 等架构边界 log）
5. 找到真路径再在那里加同源 parser

本次 v2.8.19 → v2.8.20 这一跳就是被该 SOP 第 3 步抓出来的。

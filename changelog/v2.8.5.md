# v2.8.5 (2026-04-28)

**WS BOT 模式响应速度修复 — block 流式恢复 + partial 节流上限**

## 背景

v2.7.x 时引入了一个"为防 SDK 内部 ack 队列卡死"的临时改动 ([`reply.ts:387-389`](../src/transport/bot-ws/reply.ts))：

```ts
// For non-final (block) chunks: only accumulate text, don't send.
// Sending each block via WS individually feeds the SDK's reply queue,
// where each item waits 5s for WS ack. If ack is slow, 24 blocks × 5s = 120s.
// Final delivery handles the full accumulated text in a single shot.
if (info.kind !== "final") {
  return;
}
```

**副作用**：用户体验变成「placeholder 不断闪 → 突然出完整答案」，**没有真正的流式输出**。LLM 推理 30-60 秒的场景下，用户感受到的就是"BOT 不回复"。

更糟：[`reply.test.ts`](../src/transport/bot-ws/reply.test.ts) **本来期望 block 应该按累积量发 partial replyStream**（13 个测试中 8 个失败），实现和测试已分裂数版本，但 CI 没拦下。

## 改动

### `src/transport/bot-ws/reply.ts`

1. **新增常量 `MAX_PARTIAL_REPLIES = 8`** — partial replyStream 调用次数上限，防 SDK reply queue 饱和。
2. **新增 handle 局部计数器 `partialReplyCount`** — 追踪本次 reply session 已发的 partial 次数。
3. **`info.kind !== "final"` 分支重写**（之前直接 return）：
   - 仍走 fire-and-forget 路径（`void params.client.replyStream(...).catch(...)`）— 不阻塞 deliver 主线
   - **首次 partial 发出时停 placeholder keepalive**，避免下一轮 keepalive 把 partial 内容覆盖回 "⏳ 思考中"
   - 调用 `notifyPeerActive()` 通知同 peer 的 orphan handle
   - catch ack timeout 时仍喂 `hitAckTimeoutWatchdog` 让健康监控生效
   - 超过 `MAX_PARTIAL_REPLIES` 后的 block 静默累积，由 final 一次性兜底发完整内容

### `src/transport/bot-ws/reply.test.ts`

- 给 `mockClient` 加 `isConnected: true` — `reply.ts` 在 v2.8.0 后引入了 `if (!params.client.isConnected) { fallback }` 检查，但测试 mock 没跟上，导致大量测试被强制走 fallback 路径触发误失败。

## 用户可见效果

- WeCom 中收到的回复**真正流式更新**：`占位 → 第一段 → 第一段+第二段 → ... → 完整回复`，最多 8 次中间更新
- LLM 慢的场景下，用户每 100-500ms 看到内容更新，**告别"几十秒不动一下突然全出"**
- 24 blocks × 5s = 120s 卡死风险被 `MAX_PARTIAL_REPLIES = 8` 上限 + fire-and-forget 双保护避开

## 测试

- `reply.test.ts`：**13/13 全过**（之前 8 失败 / 5 通过）
- `tsc --noEmit`：通过
- 全套 `vitest run`：176 个测试中 168 通过，8 个失败均为 v2.8.5 之前已存在的问题（`outbound.test.ts` × 4、`channel.config.test.ts` × 2、`monitor.active.test.ts` × 2），与本次改动无关

## 不变

- `PLACEHOLDER_KEEPALIVE_MS = 3000` 不变（保留 keepalive 频率）
- `MAX_KEEPALIVE_MS = 120s` 不变
- ack timeout watchdog 阈值不变（连续 8 次触发 reconnect）
- 移除的 `MiniMax-M2.7` 兜底（在 `~/.openclaw/openclaw.json` 配置层）属于运行时配置调整，不在本次代码改动里

## 兼容性
